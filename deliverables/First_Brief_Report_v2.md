## 1. Fact-Finding Techniques

*Describe the techniques used to gather requirements for this project.*

The following fact-finding techniques were used, consistent with the requirements engineering methods covered in this course:

### 1.1 Interviews
Semi-structured interviews were conducted with the project supervisor and potential system users. Interview questions were designed to explore:
- Core problems the system should address
- Target user groups and their characteristics
- Existing workflows and pain points
- Technical and security requirements

### 1.2 Document Analysis
Existing institutional documents were reviewed, including:
- Course assignment requirements and assessment criteria
- Comparable teaching assistant recruitment processes
- School administrative data management workflows

### 1.3 Questionnaires / Surveys
Structured questionnaires were distributed to target users to:
- Identify functional priorities across different user roles
- Gather quantitative data on feature preferences
- Validate assumptions from interview findings

### 1.4 Prototyping
Low-fidelity wireframe prototypes were developed and presented to stakeholders for feedback. This aligns with the iterative prototyping approach covered in Lecture 4, allowing early validation of requirements before full development.

## 2. Iteration Plan

*Describe the planned iterations (sprints) for this project.*

The project follows an Agile Scrum process over the 8-week laboratory period (Week 3–12). Development is organised into the following sprints:

| Sprint | Timeframe | Primary Goals |
|--------|-----------|---------------|
| Sprint 0 | Week 3–4 | Team formation, project topic selection, initial requirements gathering |
| Sprint 1 | Week 4–5 | Requirements analysis, Product Backlog creation, User Story writing |
| Sprint 2 | Week 5–6 | System architecture design, interface prototyping |
| Sprint 3 | Week 6–7 | Core feature development (MVP — user registration, login, data management) |
| Sprint 4 | Week 7–8 | Feature completion and internal testing |
| Sprint 5 | Week 8–9 | Integration testing and user acceptance testing |
| Sprint 6 | Week 9–10 | Documentation and demo preparation |
| Sprint 7 | Week 10–11 | Final demonstration and project wrap-up |

Each sprint follows the standard Scrum ceremonies: Sprint Planning at the start, Daily Stand-ups throughout, Sprint Review and Retrospective at the end.

## 3. Prioritisation Methods

*Describe how requirements were prioritised.*

### 3.1 MoSCoW Method
The MoSCoW prioritisation technique (Lecture 3 & 4) was used to classify all User Stories:

| Priority | Meaning | Examples in Our Backlog |
|----------|---------|------------------------|
| **Must Have** | Core functionality without which the system is unusable | User registration, login, record creation, data persistence |
| **Should Have** | Important but not safety-critical features | Role management, dashboard, data editing |
| **Could Have** | Desirable enhancements if time permits | Notifications, advanced search, data export/import |
| **Won't Have (this iteration)** | Explicitly deferred to a future release | Mobile application, multi-language support |

### 3.2 Priority Matrix
Stories were also evaluated on a two-dimensional matrix of **business value** versus **implementation complexity**. High-value, low-complexity stories were scheduled for earlier sprints.

## 4. Estimation Methods

*Describe how the effort for each requirement was estimated.*

### 4.1 Story Points with Planning Poker
Following the approach covered in Lecture 4, each User Story was estimated using **Story Points** based on the **Fibonacci sequence** (1, 2, 3, 5, 8, 13, 21). The team conducted Planning Poker sessions where each member independently estimated, then discussed discrepancies before reaching a consensus estimate.

### 4.2 Velocity-Based Planning
The team's **Velocity** (total story points completed per sprint) was estimated based on team capacity and availability. With 6 team members and an estimated 10–15 hours per person per week, each sprint is planned for approximately 20–30 story points.
